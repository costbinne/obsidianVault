# 主要目标以及纲领
## 1. 总目标

用一个“业务无关”的最小应用，完成一次**可复现的端到端交付闭环**：

> 前端 → 后端 → DB → 自动化构建/测试 → 自动部署 → 可访问 Demo

## 2. 范围边界（只做必要的）

- 功能只做 **最小 CRUD**（新增 + 查询即可）
    
- 重点是**结构、流程、可复现性**
    
- 不做复杂权限/登录、不做花哨 UI、不追求性能
    

---

## 3. 交付物（你最终必须产出这些）

### A. 系统本体

- 前端（能调用后端 API，展示数据）
    
- 后端（提供 API，读写 DB）
    
- DB（schema + 初始化方式）
    

### B. 工程化资产

- **仓库结构**清晰（monorepo 或 multi-repo 自选）
    
- **配置管理**（环境变量/配置文件，区分 dev/prod）
    
- **日志与可观测**（至少能定位错误：请求失败/DB 连接失败）
    

### C. 自动化流水线

- CI：push 后自动
    
    - build
        
    - test（最少 1 个）
        
    - 产物生成
        
- CD：满足条件后自动
    
    - 部署到一个可访问环境
        
    - 能输出访问 URL
        

### D. 文档（很关键）

- README：一页说明
    
    - 本地如何启动
        
    - 如何跑测试
        
    - CI/CD 流程是什么
        
    - 出问题怎么排查（3～5条）
        

---

## 4. 推荐实施顺序（固定顺序，避免走弯路）

1. **本地闭环**：前端能打后端，后端能读写 DB
    
2. **可复现 DB**：迁移脚本/初始化脚本（换机器也能建表）
    
3. **加 CI**：先做到“自动 build/test”
    
4. **加 CD**：自动部署到可访问环境
    
5. **补文档 + 排查手顺**：把流程固化成资产
    

---

## 5. 验收标准（完成的定义）

你完成作业的判定条件（全部满足）：

1. 新环境从零开始，仅按 README 操作 → 能跑起来
    
2. 一次 CRUD 成功（新增→刷新→能看到）
    
3. push 到远端 → CI 自动跑完且可看到日志
    
4. 主分支更新后 → 站点/服务自动更新，可通过 URL 访问
    
5. 发生错误时，你能按“分层排查顺序”定位到大类原因
    
    - OS/网络
        
    - 运行环境/中间件
        
    - 应用/代码
        
    - DB/数据

# 本次作业的目的（你为什么要做这件事）

## 目的 1：把“模糊认识”变成“可验证模型”

你现在的问题不是没学，而是**缺一个自己验证过的闭环**。  
做完一次，你脑中会形成稳定模型：

> 一个系统到底由哪些层组成、怎么连接、哪里会坏、怎么查。

---

## 目的 2：获得“工程语境里的自信”

日本现场很多工作不是写代码，而是：

- 环境/配置
    
- 版本/发布
    
- 调查/切分责任
    
- 写手顺/资料
    

你做完这套闭环，就能把很多讨论落到“证据链”。

---

## 目的 3：把 AI 变成放大器

AI 可以帮你写代码，但不能替你建立：

- 结构
    
- 分层
    
- 发布链路
    
- 排障模型
    

这次作业的产物会变成你未来任何项目的模板/骨架。

---

## 目的 4：为未来转型铺路

无论你未来偏：

- 架构/基盘
    
- 全栈
    
- DevOps
    
- 技术负责人
    

“从 0 到交付”的闭环能力都是底座。
