# AWS服务全体概要

## 一、基础网络理解

### Edge Location

- Edge Location 是什么？
- 和 Region / AZ 的区别
- CloudFront 和 Edge 的关系
- 动态请求是否也会加速
**Edge Location 是 AWS 在全球分布的“边缘节点”，本质是离用户最近的网络接入点，==主要用于加速内容分发==。它==不是用来部署业务服务器==（不能像 EC2 一样跑应用），而是==作为缓存和网络加速层==存在。**


在 AWS 的整体结构中，可以这样理解层级关系：  
Region 是一个大的地理区域（比如东京），AZ（Availability Zone）是 Region 内的多个数据中心分区，用来保证高可用，而 Edge Location 则分布在全球各地，数量远多于 AZ，作用是让用户“就近接入 AWS 网络”。


==CloudFront 是 AWS 提供的 CDN（内容分发网络）服务==，==它会利用这些 Edge Location 来工作==。换句话说，==Edge Location 是基础设施，而 CloudFront 是使用这些基础设施的服务==。用户访问 CloudFront 时，请求会先到最近的 Edge Location，如果有缓存就直接返回，如果没有，就由 Edge Location 通过 AWS 的高速骨干网络去请求源站（Origin，比如 ALB 或 EC2），再把结果返回给用户。


对于静态内容（图片、JS、CSS 等），Edge Location 可以直接缓存，因此速度提升非常明显；而对于动态请求（比如 API、动态页面），Edge 不负责生成内容，仍然需要回源到后端服务处理。但即使如此，访问路径仍然是“用户 → 就近 Edge → AWS 内部网络 → Origin”，相比用户直接跨公网访问 Region，路径更短、更稳定，因此动态请求也会获得一定程度的加速，本质是优化了网络传输，而不是减少了服务器计算时间。

### 动态请求加速

- 为什么 dynamic 页面也会变快？
- 本质是“网络路径优化”，不是计算变快

**动态请求之所以也会变快，是因为 CloudFront 把“用户到服务器”的长公网路径，拆成了“用户到就近 Edge + AWS 内部骨干网络到源站”的两段更优路径。**


具体来说，用户请求先进入最近的 Edge Location，第一段网络距离更短、握手更快；随后 Edge 到源站（Origin，如 ALB/EC2）的流量走 AWS 自有的全球骨干网络，这条网络相比公网更稳定、拥塞更少、丢包更低。同时 Edge 会复用与源站的连接（keep-alive）、使用更高效的协议（HTTP/2/HTTP/3），减少频繁建连和往返延迟（RTT）。另外，TLS 终止、头部处理等也可以在边缘完成，降低端到端开销。


需要注意的是，动态内容依然必须回源由后端计算生成，数据库查询和业务处理时间不会因为 CloudFront 而变短。因此它带来的提升主要体现在**连接建立更快、路径更稳定、网络抖动更小**，而不是“计算变快”。

## 二、VPC（Virtual Private Cloud） 基础（核心阶段）

### VPC 创建注意点
- CIDR 规划（/16 ~ /28）
- IP 不可重叠
- IPv4 vs IPv6

**VPC 创建时最核心的注意点在于 CIDR 规划、地址不重叠以及 IPv4/IPv6 的选择，这三点会直接影响后续扩展性和网络互通能力。**


首先是 CIDR 规划。创建 VPC 时必须指定一个私有 IP 地址范围（CIDR），例如 10.0.0.0/16，这个范围决定了整个 VPC 能容纳多少 IP。AWS 要求 IPv4 的 VPC CIDR 在 /16 到 /28 之间，数字越小（如 /16）代表可用 IP 越多（约 6.5 万），数字越大（如 /24、/28）则容量更小。设计时需要考虑未来要划分多个子网（public、private、不同 AZ），以及是否会扩展服务，否则网段过小会导致后期无法继续划分子网或扩容。


其次是 IP 地址不可重叠。如果你的 VPC 将来需要与其他网络通信（例如 VPC Peering、Transit Gateway、VPN、Direct Connect 连接本地机房），那么各个网络之间的 CIDR 不能有任何重叠。一旦重叠，路由无法判断流量该去哪个网络，会直接导致通信失败。因此在设计 VPC 时通常会预留和规划好不同环境（生产、测试、本地）的网段，避免冲突。


最后是 IPv4 与 IPv6 的区别。IPv4 是当前主流，地址有限，因此通常结合私网地址（10.x、172.16.x、192.168.x）和 NAT 使用；而 IPv6 地址空间极大，AWS 通常为 VPC 分配固定大小的 IPv6 地址块（如 /56），每个资源都可以拥有全球唯一地址，不需要 NAT 即可直接访问互联网。实际设计中，大多数系统仍以 IPv4 为主，但越来越多架构开始支持 IPv6 以简化网络结构和提升可扩展性。

### IP / CIDR 基础理解（你重点卡的）
- IP 地址的作用是什么？
- 为什么是四段？
- 每一段有没有意义？
- CIDR 是什么？是不是 AWS 专属？
- “前三段/后一段”有没有意义？

**IP / CIDR 的基础可以这样理解：IP 是“地址”，CIDR 是“划分规则”，两者一起决定网络结构。**

IP 地址的作用是标识网络中的一台设备，让数据知道“该发给谁”，就像现实中的门牌地址一样。没有 IP，请求就无法定位到具体的机器。

IPv4 看起来是“四段”（如 192.168.1.10），但这只是为了方便人类阅读的表示方式，-==本质是 32 位二进制，每 8 位一组转成十进制，所以变成 4 段（每段 0~255）==。因此“四段”只是显示形式，并不代表天然的层级结构。

==每一段本身并没有固定意义。现代网络中不会再说“第一段代表公司、第二段代表部门”这种规则，这些是早期的 Class A/B/C 分类，现在已经被淘汰。==真正决定哪一部分是“网络”，哪一部分是“主机”的，是 CIDR，而不是“第几段”。

CIDR（Classless Inter-Domain Routing）是一种通用的互联网标准，不是 AWS 专属。它通过“/数字”的方式表示前多少位是网络部分，例如 192.168.1.10/24 表示前 24 位是网络，剩下的是主机。这样就可以灵活划分网络大小，而不是像过去那样固定按段划分。

因此，“前三段/后一段有没有意义”的答案是：没有固定意义，只是某些常见 CIDR（比如 /24）看起来像“前三段是网络、最后一段是主机”，但如果 CIDR 改成 /16 或 /20，这种划分就会完全改变，甚至会把某一段拆开使用。

### 子网 & CIDR 实操理解
- VPC: 10.0.0.0/16 是什么意思
- 子网怎么划分（/24 等）
- 子网必须在 VPC 范围内
- CIDR 重叠是什么意思

**子网和 CIDR 的实操理解可以围绕“范围切分”来理解：VPC 定义一整块地址空间，子网是在这块空间里再细分。**

当你创建 VPC：`10.0.0.0/16` 时，意思是前 16 位固定为网络部分，也就是 `10.0`，后面的位可以变化，因此整个 VPC 的地址范围是 `10.0.0.0 ~ 10.0.255.255`，总共约 65536 个 IP。这一整块就是你在 AWS 里“拥有”的私有网络空间。

子网的本质就是从这个大范围里再切一小块出来使用。比如你可以划分出：  
`10.0.1.0/24`、`10.0.2.0/24`、`10.0.3.0/24` 等。这里的 `/24` 表示前 24 位固定（10.0.1），只剩最后 8 位可以变化，因此每个子网大约有 256 个 IP。你可以把 `/16` 看成一整块地，而 `/24` 是把这块地分成很多小区，每个小区再给具体的房子（EC2、RDS 等）分配地址。

子网必须在 VPC 范围内，这是硬性规则。也就是说你划的子网地址必须完全落在 `10.0.0.0/16` 这个范围里，比如 `10.0.1.0/24` 是合法的，但 `10.1.1.0/24` 或 `192.168.1.0/24` 就不属于这个 VPC，不能使用。可以理解为“子网是 VPC 的一部分，不能越界”。

CIDR 重叠指的是两个地址范围有交集。比如 `10.0.0.0/16` 和 `10.0.1.0/24` 是重叠的，因为后者完全包含在前者之中；或者两个 VPC 分别是 `10.0.0.0/16` 和 `10.0.0.0/24`，也属于重叠。这种情况下在做网络连接（如 VPC Peering、VPN）时会出现问题，因为路由无法判断数据应该发往哪一边。相反，如果是 `10.0.0.0/16` 和 `10.1.0.0/16`，它们范围完全不同，就不重叠，可以正常通信。
## 三、IPv4 / IPv6 / NAT（理解突破点）
### NAT 是什么
- NAT 的作用（私网 → 公网）
- NAT Gateway vs NAT Instance

**NAT（Network Address Translation，网络地址转换）本质是把私网IP转换成公网IP，从而让无法直接上网的私有资源访问互联网。**

在 VPC 中，私有子网里的实例（例如 10.0.2.10）使用的是私网 IP，这类地址在互联网中是不可路由的，外部无法识别，因此不能直接访问公网。当它需要访问外部服务（比如下载依赖、调用API）时，请求会先发送到 NAT，由 NAT 使用自己的公网IP代替该实例发出请求；返回数据再由 NAT 根据连接记录转发回原来的私网实例。整个过程对外只暴露 NAT 的公网IP，因此实现了“私网出网但不暴露自身”的效果。

NAT Gateway 和 NAT Instance 都可以实现这个功能，但方式不同。NAT Gateway 是 AWS 提供的托管服务，具备高可用、自动扩展能力，无需维护，性能稳定，是当前推荐方案；而 NAT Instance 是用一台 EC2 充当 NAT，需要手动关闭源/目的检查、自己维护高可用（例如多实例+切换）、并承担性能瓶颈，因此现在基本只在特殊场景或成本敏感时使用。

### IPv4 vs IPv6
- 为什么 IPv4 不够用
- 为什么 IPv6 不需要 NAT

**IPv4 和 IPv6 的本质差别在于“地址数量是否充足”，这直接决定了是否需要 NAT。**

IPv4 使用 32 位地址，总量大约是 43 亿个。在互联网早期这个数量看起来足够，但随着手机、云计算、IoT 等设备爆发式增长，公网 IP 很快被耗尽。为了解决地址不足的问题，引入了私网地址（如 10.x、192.168.x）和 NAT：大量设备使用私网 IP，在访问互联网时通过 NAT 共享少量公网 IP，对外看起来像是同一个地址，从而节省公网 IP 资源。

IPv6 使用 128 位地址，数量达到 2^128，几乎可以认为是无限的。每一台设备都可以拥有一个全球唯一的公网 IP，因此不需要再通过 NAT 进行地址复用。设备之间可以实现真正的端到端通信，网络结构也更加简单，不再需要额外的地址转换层。

### 设备数量问题
- 一个路由器最多能接多少设备？
- 为什么不是 43 亿？
- 私网 IP 可以重复
- NAT + 端口机制

**设备数量的问题本质不是由 IPv4 的“43亿地址”决定，而是由“私网网段大小 + NAT机制”共同决定。**

一个路由器（或者说一个私有网络）能接入多少设备，首先取决于它分配的私网 CIDR。例如常见家庭网络是 `192.168.1.0/24`，总共约256个IP，去掉网络地址和广播地址后，大约能容纳254台设备；如果使用更大的网段（如 /16），理论上可以支持几万台设备。因此设备数量的上限取决于网段大小，而不是 IPv4 的总地址数。

之所以不是“最多43亿设备”，是因为这43亿指的是**公网 IP 的数量**，而不是设备数量。现实中绝大多数设备并不直接使用公网 IP，而是使用私网 IP，通过 NAT 共享公网 IP 上网。因此设备总数可以远远超过 43 亿。

私网 IP 是可以重复使用的，这是关键点。比如你家和别人家的路由器都可以使用 `192.168.1.2`，因为这些地址只在各自的局域网内部有效，不会在互联网中冲突。只有公网 IP 才要求全球唯一。

NAT 的作用是在私网和公网之间做转换，而它不仅仅使用 IP，还会结合端口进行区分。比如多个设备通过同一个公网 IP 出网时，会被映射成不同的端口连接（例如 1.2.3.4:50001、1.2.3.4:50002），这样 NAT 就能区分不同设备的请求并正确返回数据。因此，一个公网 IP 实际可以支持成千上万的并发连接。

## 四、VPC 网络结构（开始连起来）
### Public vs Private Subnet
- 是否用不同 IP？（❌不是）
- 区别来自：
    - Route Table
    - IGW / NAT
    - 公网IP

**Public Subnet 和 Private Subnet 的区别不在于使用不同的 IP 段，而在于“是否具备访问互联网的路径”，这个能力由路由、网关和公网 IP 决定。**

在同一个 VPC 中，所有子网使用的都是同一块私有 CIDR（例如 10.0.0.0/16），无论是 Public 还是 Private Subnet，里面的实例拿到的私网 IP 本质是一样的（如 10.0.1.x、10.0.2.x），并不存在“公网子网用公网IP段、私网子网用私网IP段”的区别。

两者的核心差异来自 Route Table。Public Subnet 的路由表中会有一条 `0.0.0.0/0 → Internet Gateway (IGW)`，表示访问任意外部地址时都可以通过 IGW 出互联网；而 Private Subnet 的路由表不会指向 IGW，而是通常配置为 `0.0.0.0/0 → NAT Gateway`，或者干脆没有外网路由，因此无法被外部直接访问。

IGW 和 NAT 的作用也不同。IGW 是让资源“直接与互联网双向通信”的网关，只要实例有公网 IP，就可以被外部访问；而 NAT Gateway 只负责“让私网资源访问外网”，但不会允许外部主动访问私网实例，因此是单向的出网通道。

公网 IP 是 Public Subnet 的另一个关键因素。即使子网有 IGW 路由，如果实例没有分配公网 IP（或绑定 Elastic IP），外部仍然无法直接访问它；反过来，即使实例有公网 IP，如果子网没有 IGW 路由，也无法通信。因此是否“对外可访问”是由“路由 + 网关 + 公网 IP”共同决定的

## 五、Route Table
### Route Table 是什么
- “目标IP → 下一跳”
- local / IGW / NAT

**Route Table（路由表）本质是决定“数据要往哪里走”的规则集合，可以理解为：当目标 IP 确定后，系统根据规则选择对应的下一跳（Next Hop）。**

每一条路由规则都由两部分组成：一是目标地址范围（Destination，例如某个 CIDR），二是对应的目标（Target，也就是下一跳）。当实例发起请求时，AWS 会根据目标 IP 去匹配路由表中的规则，并选择“最精确匹配”的那一条，然后把流量发往对应的下一跳。

在 VPC 中最常见的几种 Target 有三种。`local` 表示 VPC 内部通信，也就是只要目标 IP 在 VPC 的 CIDR 范围内，流量就在内部直接转发，不会离开 AWS 网络；`Internet Gateway（IGW）` 表示把流量发送到互联网，用于 Public Subnet 对外通信；`NAT Gateway` 则用于 Private Subnet，让私网实例通过 NAT 出网访问互联网，但不会暴露自身。

通常一张路由表里会有两类典型规则：一条是指向 VPC 自身 CIDR 的 `local` 规则（例如 10.0.0.0/16 → local），用于内部通信；另一条是 `0.0.0.0/0` 的默认路由，表示“所有不在本 VPC 内的流量”。这条默认路由会根据子网类型指向 IGW（Public Subnet）或 NAT Gateway（Private Subnet），从而决定是否以及如何访问互联网。

### 0.0.0.0/0 的意义
- 表示所有地址
- 默认出口

**0.0.0.0/0 在路由表中的含义是“匹配所有 IPv4 地址”，因此通常被用作默认出口（Default Route）。**

从 CIDR 的角度看，“/0”表示前 0 位是固定的，也就是说没有任何限制，任何 IP 地址都能匹配这条规则，因此 0.0.0.0/0 就代表“全世界所有 IPv4 地址”。当一个目标 IP 不属于 VPC 内部网段（例如不在 10.0.0.0/16 里）时，请求就会匹配到这条规则。

在实际使用中，0.0.0.0/0 通常作为“兜底规则”。路由系统会优先匹配更具体的网段（比如 /24、/16），只有当没有更精确匹配时，才会使用这条最宽泛的规则。因此它并不是因为“写在最后才生效”，而是因为它的匹配范围最大、优先级最低。

在 AWS VPC 中，这条规则的目标（Target）决定了子网的网络性质。如果 0.0.0.0/0 指向 Internet Gateway（IGW），则表示流量可以直接访问互联网，这样的子网就是 Public Subnet；如果指向 NAT Gateway，则表示只能通过 NAT 出网，这样的子网是 Private Subnet；如果没有这条规则，则该子网无法访问互联网。

### 路由匹配规则
- 是否按顺序执行？（❌不是）
- 最长前缀匹配（Longest Prefix Match）

**路由表的匹配规则不是按顺序执行的，而是遵循“最长前缀匹配（Longest Prefix Match）”原则。**

当一条请求到达时，系统会拿目标 IP 去匹配路由表中的所有规则，而不是从上到下逐条判断。只要规则的 CIDR 范围能够覆盖这个目标 IP，就算“匹配成功”，然后在所有匹配的规则中，选择“最具体、最精确”的那一条来决定流量走向。

所谓“最长前缀”，指的是 CIDR 中“/后面的数字越大，匹配范围越小、越精确”。例如 `/24` 比 `/16` 更具体，`/16` 又比 `/0` 更具体。因此优先级实际上是：  
更具体的网段（/24） > 较大的网段（/16） > 默认路由（/0）。

举个典型例子：如果路由表中有  
10.0.1.0/24 → NAT  
10.0.0.0/16 → local  
0.0.0.0/0 → IGW  
当目标是 10.0.1.5 时，这三条规则其实都能匹配，但最终会选择 `/24`，因为它最精确；如果目标是 10.0.2.5，则会匹配 `/16`；如果目标是公网地址（如 8.8.8.8），才会走 `/0` 这条默认路由。

因此，0.0.0.0/0 之所以作为“默认出口”，不是因为它写在最后，而是因为它最不具体，只在没有更精确匹配时才会被选中。

## 六、Internet Gateway / NAT
### IGW（Internet Gateway）
- Public Subnet 的核心条件
- 直接对外通信

**Internet Gateway（IGW）是连接 VPC 与互联网的网关，它的存在使得资源可以与公网进行双向通信，是 Public Subnet 成立的核心条件之一。**

IGW 本质上是一个附加在 VPC 上的组件，它提供了从 VPC 到互联网的出口，同时也允许来自互联网的流量进入 VPC。但它本身并不会自动让资源上网，必须配合路由表使用：当子网的 Route Table 中存在 `0.0.0.0/0 → IGW` 这条规则时，才表示该子网可以将流量发送到互联网。

==是否是 Public Subnet==，并不是由子网名称或 IP 决定，而是取决于==两个条件：第一，路由表中必须有指向 IGW 的默认路由；第二，子网内的实例需要拥有公网 IP（或绑定 Elastic IP）==。只有同时满足这两点，==外部用户才能访问该实例，该子网也才真正具备“对外公开”的能力==。

==IGW 支持的是“直接对外通信”，也就是双向的：一方面，子网内的实例可以主动访问互联网；另一方面，互联网中的客户端也可以通过公网 IP 主动访问该实例（前提是安全组等允许）==。这也是它与 NAT Gateway 的本质区别，后者只支持私网实例单向出网，不允许外部主动访问。

### NAT Gateway
- Private Subnet 出网方式
- “只能出不能进”

**NAT Gateway 是 Private Subnet 访问互联网的标准方式，它的核心作用是让私网实例能够“对外访问”，但不会被外部主动访问。**

在 Private Subnet 中，实例只有私网 IP（如 10.0.2.10），这种地址在互联网中不可路由，因此无法直接访问公网。通过在公有子网中部署 NAT Gateway（并为其绑定公网 IP），再在私有子网的路由表中配置 `0.0.0.0/0 → NAT Gateway`，私网实例发出的请求会先到 NAT，由 NAT 用自己的公网 IP 替换源地址后再访问互联网。返回流量再根据连接记录被转发回原来的私网实例。

所谓“只能出不能进”，指的是 NAT Gateway 只会为“已经发起的请求”建立映射并转发返回流量，而不会接受来自互联网的主动连接请求。外部无法直接通过 NAT Gateway 访问私网实例，因此它天然具备隔离和保护内部资源的作用。这与 Internet Gateway 不同，IGW 支持双向通信，而 NAT Gateway 只提供单向的出站能力。

## 七、安全控制（后半部分）

### Security Group（SG）
- 是什么（实例级防火墙）
- Inbound / Outbound
- 默认规则（入拒绝 / 出允许）
- 端口是什么（80 / 443 / 22）
- 0.0.0.0/0 含义
- Stateful（有状态）

**Security Group（SG）是绑定在实例（如 EC2）上的虚拟防火墙，用来控制该实例的网络访问规则。**

它通过两类规则来工作：Inbound（入站）和 Outbound（出站）。Inbound 决定“谁可以访问这台机器”，例如允许某个 IP 访问 80 端口；Outbound 决定“这台机器可以访问谁”，例如允许访问互联网或其他服务。

默认情况下，Security Group 的策略是：入站全部拒绝，出站全部允许。这意味着新建实例默认不能被外部访问，但可以主动访问外部资源，从安全角度来说是“默认关闭入口、开放出口”。

端口（Port）表示一台机器上具体的服务入口。同一个 IP 上可以同时运行多个服务，通过不同端口区分，例如 80 对应 HTTP（网站）、443 对应 HTTPS（加密网站）、22 对应 SSH（远程登录）。在 SG 中，你就是通过“端口 + 协议 + 来源”来定义访问规则。

0.0.0.0/0 表示所有 IPv4 地址，也就是“任何来源”。如果你在入站规则中配置“80端口 + 0.0.0.0/0”，就表示允许全世界访问你的 Web 服务；如果用于 SSH（22端口），则意味着任何人都可以尝试连接，通常是不安全的做法。

Security Group 的一个非常重要特性是“有状态（Stateful）”。这意味着只要允许某个方向的请求，返回流量会被自动允许，无需额外配置。例如你允许出站访问某个网站，那么该网站返回的数据会自动放行，即使没有对应的入站规则。


### NACL（网络ACL）
- 子网级防火墙
- Stateless（无状态）
- 可以 DENY
- 按顺序执行
- 默认规则 vs 自定义规则

**NACL（Network ACL）是作用在子网（Subnet）层级的防火墙，用来控制整个子网进出流量的允许与拒绝。**

与 Security Group 不同，NACL 是无状态（Stateless）的，这意味着入站和出站是完全独立的：如果你允许某个请求进入，还必须单独写一条规则允许返回流量，否则响应会被拦截。因此在设计规则时需要成对考虑“请求和返回”。

NACL 支持显式的拒绝（DENY）规则，这是它区别于 Security Group 的重要特性。你可以明确阻止某些 IP 或端口的访问，而不仅仅是“默认不允许”。同时，NACL 的规则是按编号顺序执行的，从小到大依次匹配，一旦命中某条规则（允许或拒绝）就会停止继续匹配，因此规则顺序会直接影响结果。

在默认行为上，AWS 提供的默认 NACL 是“全部允许”（入站和出站都允许），而一旦你创建自定义 NACL，默认就是“全部拒绝”，需要你手动添加允许规则。这一点在考试中经常考察。

### SG vs NACL 对比（重点）
- SG：实例级 + 有状态
- NACL：子网级 + 无状态 + 可拒绝

**Security Group（SG）和 NACL 的区别本质在于“作用范围、状态机制以及控制能力”，两者共同构成 VPC 的安全体系。**

Security Group 作用在实例（如 EC2）层面，是实例级防火墙；NACL 作用在子网层面，是子网级防火墙。数据流量进入一台实例时，必须同时通过 NACL（子网边界）和 SG（实例边界）的检查，可以理解为先过“楼层门禁”，再过“房间门锁”。

在状态机制上，SG 是有状态（Stateful）的，只要允许某个方向的请求，返回流量会自动放行，因此规则配置更简单；而 NACL 是无状态（Stateless）的，入站和出站必须分别配置，请求和响应都需要显式允许，否则会被拦截。

在控制能力上，SG 只能定义“允许（ALLOW）”规则，未被允许的流量默认拒绝；而 NACL 不仅可以允许，还可以显式拒绝（DENY），并且按规则编号顺序匹配，这使得它更适合做粗粒度的访问控制或安全隔离。

##  9. VPC和外部的接线
### on-premises（本地环境）是什么
本地环境指的是企业自己拥有的服务器和数据中心（比如公司机房里的服务器），与部署在 AWS 云上的资源相对应。

### 链接on-premises 和 VPC的方式
#### 1.Direct Connect
Direct Connect 是专线连接，相当于企业机房和 AWS 之间拉了一条“专用光纤”，不走互联网，因此延迟更低、稳定性更高、带宽更可控，适合对网络质量要求很高的系统（例如金融系统）。另外通过 Direct Connect Gateway，还可以实现一个本地环境连接多个 VPC。

#### 2. VPN（Virtual Private Network）
VPN（Virtual Private Network）则是通过互联网建立加密隧道连接 AWS 和本地环境。它分为两种：==Site-to-Site VPN== 和 ==Client VPN==。Site-to-Site VPN 是“公司网络 ↔ AWS VPC”的连接方式，适用于整个机房互通；Client VPN 是“个人设备 ↔ AWS VPC”，适用于开发者或员工远程接入云环境。相比 Direct Connect，VPN 成本更低、部署更快，但因为走公网，稳定性和性能不如专线。

##### Site-to-Site VPN中的加密
使用的技术：

IPsec（Internet Protocol Security）是一种用于在网络层对数据进行加密和认证的协议，主要作用是在不安全的公网（互联网）上建立“安全通信隧道”。

AWS 的 Site-to-Site VPN 中，IPsec 的作用是：

把“公司本地机房 ↔ AWS VPC”之间的通信数据进行加密，然后通过互联网传输。

完整性保护（Integrity）  
防止数据被篡改

### VGW
![[Pasted image 20260412215226.png]]
Virtual Private Gateway（VGW）是用于“VPC ↔ 本地环境（on-premises）”连接的网关，它是部署在 AWS 侧的 VPN 入口。使用 Site-to-Site VPN 或 Direct Connect 时，需要先在 VPC 上创建并附加 VGW，然后本地环境通过 VPN（IPsec）或专线（DX）连接到这个 VGW，从而实现公司机房与 AWS VPC 的私有网络互通。可以理解为：VGW 是 AWS 这边的“门口”，专门用来接外部网络进来。

在连接方式上，Direct Connect 和 Site-to-Site VPN 都可以用于连接本地环境，但两者特点不同：Direct Connect 是专线连接，稳定性高、延迟低，但成本高、部署慢；Site-to-Site VPN 则是通过互联网建立 IPsec 加密隧道，成本低、部署快，但网络质量依赖公网，稳定性略差。因此实际架构中常见的是“VPN 先接入，后续再升级为 Direct Connect”。

### VPC Peering
VPC Peering 则是用于“VPC ↔ VPC”之间的私有连接。由于每个 VPC 是独立的网络，默认无法互通，通过 Peering 可以让两个 VPC 之间直接在 AWS 内部网络通信，不经过互联网，延迟低且安全。需要注意的是，只有当两个 VPC 的 CIDR 不重叠时才能建立 Peering，而且它不支持“中转”（非传递性），即 A 连 B、B 连 C，并不代表 A 可以访问 C。

不同AWS account下的VPC 也可以做peering，但是需要注意network address 需要不一致or不重复

![[Pasted image 20260412215631.png]]

### AWS Transit Gateway
**Transit Gateway 是 AWS 提供的“网络中枢（Hub）服务”，用于把多个 VPC 和本地环境（on-premises）统一连接在一起，从而简化网络架构和管理。**

在没有 Transit Gateway 的情况下，如果你有多个 VPC 需要互通，通常要使用 VPC Peering，一个 VPC 要和另一个 VPC 单独建立连接。当 VPC 数量增加时，会变成“网状连接”（Mesh），例如 3 个 VPC 需要 3 条连接，5 个 VPC 需要 10 条连接，配置和管理会变得非常复杂。

Transit Gateway 的作用就是把这种“点对点连接”变成“中心化连接”。你只需要把所有 VPC 和本地环境都连接到 Transit Gateway，然后它作为一个“中转中心”负责转发流量，实现各个网络之间的互通。

#### 能连接什么？

Transit Gateway 可以连接：

- 多个 VPC
- VPN（Site-to-Site）
- Direct Connect
- 本地机房（on-premises）

返回多个IP（粗粒度分流）

第二层：ELB内部

即使选了一个IP：

ELB节点 → 再分流到EC2

DNS只是第一层分流，真正的负载均衡在ELB内部完成

DNS → 粗分流  
ELB → 精细分流（健康检查、负载）

DNS → 返回多个ELB IP  
↓  
浏览器选一个IP  
↓  
进入某个ELB节点  
↓  
ELB根据负载分到某台EC2


### 外部ELB 与 内部ELB
ELB 可以分为两种运行方式：

- **Internet-facing（面向互联网）**
- **Internal（内部使用）**

#### #Internet-facing 
- ELB 本身部署在 **Public Subnet（公有子网）**
- 因此可以被互联网访问

关于后端（Backend Instance）：

- 可以放在 **Public Subnet**
- 也可以放在 **Private Subnet**
 
但推荐：
- 放在 Private Subnet（提升安全性）
##### 放Private Subnet理由
当 EC2 放在 Public Subnet 时，如果有公网 IP，用户可以直接访问这台机器。这样会带来问题：用户可以绕过 ELB，直接访问服务器，安全控制失效。

当 EC2 放在 Private Subnet 时，没有直接连接互联网的路径（没有 #IGW + 公网 IP），因此用户无法直接访问这台 EC2。  
用户只能通过 ELB 访问系统。

所以 Private Subnet 的作用是：  
限制访问路径，让 ELB 成为唯一入口。
#### #Internal
**Internal 的情况**

- 仅用于 **VPC 内部**
- 或 **on-premises（本地环境）**

特点：

- 不对互联网开放
- 只能内部访问


### AWS Certificate Manager（ #ACM ）
ACM 用于：

- 统一管理 SSL 证书
- 支持多个 AWS 服务

ACM 可以：

- 新建证书（发放 SSL）
- 导入已有证书

主要使用场景：

- ELB
- CloudFront
- API Gateway

用途：

- 实现 HTTPS 通信
#### 解释
ACM 是用来管理 SSL/TLS 证书的。  
证书的作用是支持 HTTPS，也就是加密通信。

它通常用在 ELB 上，而不是 EC2 上。

用户访问流程是：

用户 → HTTPS → ELB（使用 ACM 证书） → EC2

这里的关键点是：  
加密发生在“用户到 ELB”这一段。

用户（Internet）  
→ ELB（Public Subnet，使用 ACM 提供 HTTPS）  
→ EC2（Private Subnet）

这个结构实现了两件事：

第一，EC2 不对公网开放（通过 Private Subnet 实现）  
第二，用户访问是加密的（通过 ACM + HTTPS 实现）
### 总结ELB
ELB 分为 Internet-facing 和 Internal 两种类型，前者用于对外公开访问，后者用于内部通信；ACM 用于统一管理 SSL 证书，并与 ELB 等服务结合实现 HTTPS 通信。

## Amazon CloudFront #AmazonCloudFront
CloudFront 是 AWS 提供的 #CDN （Content Delivery Network）服务。  
它通过分布在全球的 Edge Location（边缘节点）向用户分发内容。

使用 CloudFront 时，用户不会直接访问源站（Origin），而是访问离自己最近的边缘节点。  
如果该节点有缓存内容，就直接返回；如果没有，则向源站获取后再返回给用户，并缓存一份。

其结果是：  
减少网络传输距离，降低延迟，提高访问速度。
### 基本结构

请求路径：

用户  
→ 最近的 Edge Location  
→（如果未命中缓存）→ Origin（源站）  
→ 返回内容并缓存

### Origin（源站）

CloudFront 可以连接的源站包括：

- ELB
- EC2
- S3

Origin 的定义是：  
提供原始内容的服务（CloudFront 负责缓存和分发）。
### 主要功能

#### SSL 加密通信

CloudFront 支持 HTTPS。  
可以使用 SSL 证书进行通信加密。

支持两种方式：

- 使用 AWS 提供的证书
- 导入自定义证书

此外，CloudFront 到后端 AWS 服务之间也可以使用 HTTPS。

#### 签名 URL（Signed URL）

可以生成带有效期的 URL。  
只有持有该 URL 的用户，且在有效时间内，才能访问内容。

用途：

- 付费内容控制
- 限时访问控制

#### 自定义错误页面

当源站返回错误（如 404、500）时，  
可以显示自定义的错误页面，而不是默认错误。

#### 地域限制（ #GeoRestriction ）

可以根据用户的地理位置：

- 允许访问
- 拒绝访问

用于内容版权控制或地区限制。

### 流媒体分发

支持视频、音频的流式传输（Streaming）。  
不仅限于静态文件分发。

### 性能和特点

- 利用全球 Edge Location 分发内容
- 降低延迟（Low Latency）
- 提高吞吐（高性能）
- 减轻源站压力（缓存）

### 试验对策部分（原文要点整理）

CloudFront 的重要点：

- 可以缓存来自 ELB、EC2、S3 的内容
- 可以提升 Web 服务性能
- 支持全球分发
- 可以通过 Signed URL 控制访问
- 可以通过地域限制控制访问范围

## Amazon Route 53 #AmazonRoute53
Amazon Route 53 是 AWS 提供的高可用 DNS 托管服务。  
在主机区域（Hosted Zone）的设置中，分为用于外部的 **Public Hosted Zone（公有托管区域）** 和用于 VPC 内的 **Private Hosted Zone（私有托管区域）**。

Public Hosted Zone 用于公开 Web 服务的域名解析。  
Private Hosted Zone 用于公司内部系统等，不经过互联网的通信中的名称解析。

---

Route 53 还提供一种称为 **ALIAS 记录** 的独特 DNS 功能。  
ALIAS 记录可以将 ELB 或 CloudFront 的端点作为目标进行指定。

与 CNAME 记录不同，可以在 **Zone Apex（根域名）** 上使用。  
因此可以像 A 记录一样使用。

---

此外，Route 53 不仅具备 DNS 功能，还可以使用 **路由策略（Routing Policy）** 来控制通信。  
在创建记录时，可以设置 Route 53 如何响应 DNS 查询。

---

### 主要路由策略

---

#### レイテンシーベースルーティング（延迟路由）

根据延迟最低的资源进行路由。

---

#### 加重ルーティング（加权路由）

为多个资源设置权重，按比例分配流量。

---

#### 位置情報ルーティング（地理位置路由）

根据客户端所在位置，将流量路由到地理上较近的资源。

---

#### フェイルオーバールーティング（故障切换路由）

对资源进行健康检查，并将流量路由到可用的资源。

---

#### シンプルルーティング（简单路由）

返回单一资源记录。

---

#### 地理的近接性ルーティング（地理邻近路由）

根据用户与资源之间的地理距离进行路由。

---

#### 複数回答ルーティング（多值应答路由）

从多个资源中随机返回健康的记录作为 DNS 响应。

---

### 试验对策

需要掌握 Route 53 的路由策略。  
特别是：

- 使用故障切换路由实现高可用架构
- 使用延迟路由实现高性能网络

---

### 补充

Route 53 不仅可以使用单一的路由策略，  
还可以组合多个路由策略进行设置。
